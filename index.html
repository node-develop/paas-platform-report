★ Deploy Control Bonus (max +2.0)¶

Bonus for control over user deployments. Evaluated across 5 axes (scale 1–5):

  1. RBAC granularity — number of roles, levels, customization
  2. Pipeline control — who can deploy, require approval, protected branches
  3. Audit trail — completeness of audit log, retention, export
  4. Rollback / preview — quality of rollback and preview environments
  5. SSO enforcement — ability to require all logins via SSO/Okta

Formula: Deploy bonus = (average of 5 axes) / 5 × 2

Total: 46 requirements + deploy control bonus (max 2.0) → maximum 48 points.

Okta SSO Migration Checklist¶

For an enterprise team implementing Okta SSO on PaaS:

  1. Inventory current access
  2. List all users and their roles on the current platform
  3. Map Okta groups to future PaaS roles

  4. Identify protected environments / production access

  5. Platform selection by Okta readiness

  6. Verify: does the platform support SAML/OIDC for Okta
  7. Verify: is there SCIM for automatic provisioning
  8. Verify: is there group-to-role mapping

  9. Verify: can SSO-only login be enforced

  10. Okta configuration

  11. Create OIDC/SAML Application in Okta Admin Console
  12. Configure groups for each PaaS role
  13. Assign users to groups

  14. Test SSO connection

  15. Platform configuration

  16. Enter Client ID / Client Secret (OIDC) or Entity ID / SSO URL (SAML)
  17. Configure team_role mapping or equivalent
  18. Enable SSO enforcement

  19. Test with a test user

  20. Full migration

7. Git Strategy¶

Recommended Approach for Enterprise Teams¶

main (production)
  └── staging (pre-prod, auto-deploy)
       └── feature/* (PR → preview environment)

Key principles:

  1. Protected branches: main and staging — protected. Deploy only via PR.
  2. Preview environments: every PR gets an isolated environment (Northflank, Vercel, Render, Railway — all supported).
  3. Approval gates: PR to staging/main requires code review + admin approval.
  4. Rollback: the platform must support instant rollback (Vercel, Northflank, Railway — best).
  5. Audit: all deployments are logged with user attribution (via SSO identity).

Appendix B: Frequently Asked Questions (FAQ)¶

Q: Why does the report cover 15 platforms and not only enterprise-grade ones? A: For completeness, the report includes self-hosted (Coolify, CapRover, Dokploy), budget (Hostinger), serverless/BaaS (Firebase), and niche (Replit) solutions. This allows the choice of an enterprise platform to be justified before stakeholders.

Q: Why does Northflank lead despite the absence of ISO 27001? A: Northflank scores highly due to: (1) BYOC — data stays in your VPC, (2) Kata Containers — microVM isolation, (3) full RBAC + SSO + audit, (4) built-in CI/CD + managed DB + GPU. ISO 27001 is important but just one of 46 criteria.

Q: Render or Vercel for a full-stack application? A: Render — for full-stack (frontend + backend + DB). Vercel — for frontend/serverless only. Render supports persistent storage, managed DB, Docker containers. Vercel is serverless functions, edge-optimized.

Q: Can Fly.io be used for enterprise? A: Technically — yes (good infrastructure, global network). But for enterprise with Okta requirements — no. No SAML, no OIDC, no audit logs. Suitable for developer-first teams without strict enterprise requirements.

Q: How to choose between managed PaaS and BYOC? A: Managed PaaS (Render, Railway, Vercel) — less operations overhead, fast start, but vendor lock-in and limited control. BYOC (Northflank, Qovery, Porter) — data in your VPC, compliance-friendly, but higher cloud costs and complexity.

Q: Which platform is best for a startup growing into enterprise? A: Railway — best path from startup ($5/mo) to enterprise ($2000+). Or DigitalOcean App Platform — free SSO Okta from day one, modular scaling.

Q: Why is Deploy Control Bonus needed? A: Deploy Control is a meta-requirement that aggregates 5 key aspects of enterprise control (RBAC, pipeline, audit, rollback, SSO). The bonus of up to +2.0 points differentiates platforms with good baseline coverage but different levels of control.

Q: Why does DigitalOcean rank high in the Okta scenario but not overall? A: DO App Platform is the only platform with free Okta SSO for all plans. This is unique. However, in terms of infrastructure capabilities (no BYOC, no GPU, limited autoscaling), it falls behind Northflank and Qovery.

Q: How to interpret the ⚡ symbol? A: ⚡ means "the feature exists but requires a paid plan or add-on." This is not the same as ⚠️ (partial implementation). ⚡ receives 0.75 points because the feature is fully functional but comes at an additional cost.

★ Deploy Control Bonus (max +2.0)¶

Бонус за контроль над пользовательскими деплоями. Оценивается по 5 осям (шкала 1–5):

  1. RBAC-гранулярность — количество ролей, уровней, кастомизация
  2. Pipeline control — кто может деплоить, require approval, protected branches
  3. Audit trail — полнота аудит-лога, ретеншн, экспорт
  4. Rollback / preview — качество rollback и preview-сред
  5. SSO enforcement — возможность обязать вход через SSO/Okta

Формула: Deploy bonus = (среднее 5 осей) / 5 × 2

Итого: 46 требований + бонус deploy control (max 2.0) → максимум 48 баллов.

Ключевые преимущества: - Полная интеграция с экосистемой Google Cloud (IAM, Monitoring, Logging, Secret Manager, Cloud Armor) - Комплексный комплаенс: SOC 1/2/3, ISO 27001/27017/27018, HIPAA BAA (для подходящих сервисов) - Serverless автоскейлинг всех сервисов — zero-config, от 0 до N - Отличный DX: Firebase Console, CLI, Admin SDK (Node.js, Python, Go, Java, C#) - Firebase Hosting: preview channels для PR, мгновенный rollback, глобальный CDN - Полная поддержка IaC: Terraform + Pulumi провайдеры, Firebase CLI

Okta-интеграция: - OK1: ⚡ Интеграция с Okta через Google Cloud Identity / Workforce Identity Federation (SAML 2.0 + OIDC). Требует Google Workspace или Cloud Identity Premium. - OK2: ⚡ SCIM-провижининг из Okta в Google Workspace (готовый коннектор). Автоматизированное управление жизненным циклом пользователей. - OK3: ⚠️ Многоступенчатый маппинг: Okta groups → Google Groups → GCP IAM roles → Firebase roles. Нет прямого маппинга.

Слабые стороны: - Нет BYOC — привязка к Google Cloud; нет мультиоблачного варианта - Только serverless-модель: нет persistent volumes, нет традиционных контейнеров, нет GPU - Непредсказуемое ценообразование: usage-based с риском bill shock при всплесках трафика - Нет нативных workflow одобрения деплоев — зависит от внешнего CI/CD - Интеграция с Okta непрямая — требует Google Cloud Identity/Workspace как посредника - RBAC только на уровне проекта — нужны отдельные GCP-проекты для изоляции окружений - Не все сервисы HIPAA-eligible (Firebase Analytics исключён)

Лучше всего подходит для: Команд, строящих serverless-приложения (web/mobile), которым нужны сильные compliance-сертификации и которые уже используют или готовы использовать экосистему Google Cloud. Особенно подходит для real-time приложений (Firestore, Realtime Database) с требованиями автоскейлинга.

7. Git-стратегия¶

Рекомендуемый подход для enterprise-команды¶

main (production)
  └── staging (pre-prod, auto-deploy)
       └── feature/* (PR → preview environment)

Ключевые принципы:

  1. Protected branches: main и staging — protected. Деплой только через PR.
  2. Preview environments: каждый PR получает изолированное окружение (Northflank, Vercel, Render, Railway — все поддерживают).
  3. Approval gates: PR в staging/main требует code review + admin approval.
  4. Rollback: платформа должна поддерживать instant rollback (Vercel, Northflank, Railway — лучшие).
  5. Audit: все деплои логируются с привязкой к пользователю (через SSO identity).

Q: Почему в отчёте 15 платформ, а не только enterprise-grade? A: Для полноты картины включены self-hosted (Coolify, CapRover, Dokploy), budget (Hostinger), serverless/BaaS (Firebase), и нишевые (Replit) решения. Это позволяет обосновать выбор enterprise-платформы перед stakeholders.

Q: Почему Northflank лидирует, несмотря на отсутствие ISO 27001? A: Northflank набирает высокие баллы за: (1) BYOC — данные остаются в вашем VPC, (2) Kata Containers — микро-ВМ изоляция, (3) полный RBAC + SSO + audit, (4) встроенный CI/CD + managed DB + GPU. ISO 27001 — важный, но один из 46 пунктов.

Q: Render или Vercel для full-stack приложения? A: Render — для full-stack (frontend + backend + DB). Vercel — только для frontend/serverless. Render поддерживает persistent storage, managed DB, Docker containers. Vercel — serverless functions, edge-оптимизирован.

Q: Можно ли использовать Fly.io для enterprise? A: Технически — да (хорошая инфраструктура, глобальная сеть). Но для enterprise с Okta-требованием — нет. Нет SAML, нет OIDC, нет аудит-логов. Подходит для developer-first команд без строгих enterprise-требований.

Q: Как выбрать между managed PaaS и BYOC? A: Managed PaaS (Render, Railway, Vercel) — меньше operations overhead, быстрый старт, но vendor lock-in и ограниченный контроль. BYOC (Northflank, Qovery, Porter) — данные в вашем VPC, compliance-friendly, но выше стоимость облака и complexity.

Q: Какая платформа лучше для стартапа, растущего в enterprise? A: Railway — лучший путь от стартапа ($5/мес) к enterprise ($2000+). Или DigitalOcean App Platform — бесплатный SSO Okta с первого дня, модульное масштабирование.

Q: Зачем нужен Deploy Control Bonus? A: Deploy Control — meta-требование, которое агрегирует 5 ключевых аспектов enterprise-контроля (RBAC, pipeline, audit, rollback, SSO). Бонус до +2.0 баллов позволяет дифференцировать платформы с хорошим базовым покрытием, но разным уровнем контроля.

Q: Почему DigitalOcean занимает высокое место в Okta-сценарии, но не в общем? A: DO App Platform — единственная платформа с бесплатным Okta SSO для всех планов. Это уникально. Однако по инфраструктурным возможностям (нет BYOC, нет GPU, ограниченный autoscaling) она уступает Northflank и Qovery.

Q: Как интерпретировать символ ⚡? A: ⚡ означает «функция существует, но требует платного плана или add-on». Это не то же самое, что ⚠️ (частичная реализация). ⚡ получает 0.75 балла, так как функция полноценна, но за дополнительную плату.

✅ Полная поддержка (1.0 балл)¶

⚡ Платно / Enterprise only (0.75 балла)¶

⚠️ Частично / workaround (0.5 балла)¶

Функция существует, но с ограничениями, неполной реализацией или через workaround. - Пример: Fly.io P1 RBAC — базовые org-level роли, нет гранулярности. - Пример: Porter OK1 Okta — SSO не задокументировано, предположительно через K8s OIDC.

🔧 DIY / ручная настройка (0.25 балла)¶

Функция не предоставляется платформой, но может быть реализована самостоятельно. - Пример: Hostinger I1 Контейнеры — VPS без Docker из коробки, нужна ручная установка. - Пример: Hostinger S3 Network Policies — iptables нужно настраивать вручную.

❌ Нет (0 баллов)¶

Функция отсутствует и не может быть разумно реализована. - Пример: Fly.io OK1 Okta — нет поддержки Okta, SAML, OIDC. Только Google/GitHub. - Пример: CapRover P1 RBAC — single admin, нет ролевой модели.
